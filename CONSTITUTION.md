# CONSTITUTION

Unbreakable rules for the TJ Monserrat personal website. These rules are derived from the finalized specification documents (`specs/01` through `specs/07`) and represent hard constraints that must never be violated by any implementation, deployment, or operational change.

Spec identifiers (e.g., `SEC-004`, `AD-004`) reference the originating specification section for traceability.

---

## 1. Security Absolutes

### 1.1 Error Response Security (SEC-004)

- **THE SYSTEM SHALL NEVER return HTTP `500` to clients.** Internal errors are masked as `404`.
- **THE SYSTEM SHALL NEVER expose internal error details, stack traces, or system information in API responses.**
- All internal errors SHALL be logged with full context (request ID, timestamp, path, client IP, error type, message, stack trace, Cloud Run instance ID) to Cloud Logging.

### 1.2 Secrets and Credentials

- **No hardcoded secrets.** All secrets are managed via environment variables or GCP Secret Manager.
- **No service account keys (JSON key files) in CI/CD pipelines.** All CI/CD authentication uses Workload Identity Federation with short-lived OIDC tokens (SEC-010, SEC-012, SEC-014).
- Service account keys SHALL NOT be committed to any repository.
- The JWT secret for `POST /t` SHALL be rotatable via environment variable changes on both the frontend build and backend deployment (SEC-003A).

### 1.3 Service Account Least Privilege (SEC-013)

- **No service account SHALL have `roles/owner` or `roles/editor`.** Only specific, granular IAM roles are permitted.
- **THE SYSTEM SHALL NOT use default service accounts** (e.g., Compute Engine default SA, App Engine default SA) for any component. Each component has a dedicated service account.
- Service accounts SHALL NOT have cross-component access unless explicitly documented in the spec.
- Cloud Run SHALL NOT have write access to Firestore Native. Only the `sync-article-embeddings` Cloud Function has write access (SEC-011).
- No service account other than Cloud Run and `sync-article-embeddings` SHALL access the Vertex AI Embeddings API (SEC-011).

### 1.4 Breadcrumb Sensitivity (BE-BREADCRUMB)

- **THE SYSTEM SHALL NEVER record sensitive data in breadcrumbs**: no JWT tokens, no API keys, no passwords, no full IP addresses, no raw user input values, no database document contents.
- Database query breadcrumbs SHALL record filter **keys** (field names) but NOT filter **values**.
- Client IPs in breadcrumbs SHALL be truncated (same rules as SEC-007).

## 2. Data Privacy Absolutes (SEC-007)

### 2.1 IP Address Anonymization

- IP addresses SHALL be truncated **before** emitting structured log entries:
  - **IPv4**: Zero the last octet (e.g., `203.0.113.42` → `203.0.113.0`).
  - **IPv6**: Zero the last 80 bits.
  - **IPv4-mapped IPv6**: Treat as IPv4 for truncation (e.g., `::ffff:203.0.113.42` → `::ffff:203.0.113.0`).
- **THE SYSTEM SHALL NOT store full IP addresses** in application-level logs, Firestore, or any application-managed data store.
- **Exception — Cloud Armor LB Logs**: Infrastructure-level load balancer logs contain full IPs (generated by Google Cloud before requests reach the backend). These are retained for a maximum of **90 days** and are accessible only to the website owner.
- **Exception — Rate Limit Offenders (DM-009)**: Full IPs stored because Cloud Armor bans are enforced on exact IPs. Records retained for a maximum of **90 days** (no active ban).

### 2.2 No Tracking Infrastructure

- **THE SYSTEM SHALL NOT use cookies, session identifiers, user accounts, or device fingerprinting.**
- **THE SYSTEM SHALL NOT share any collected data with third parties.**
- No third-party analytics. All visitor tracking is self-hosted via `POST /t` → Cloud Logging → BigQuery → Looker Studio.

### 2.3 Data Retention

- BigQuery data SHALL be automatically deleted after **2 years** via dataset-level table expiry.
- Cloud Armor LB logs (`cloud_armor_lb_logs`) SHALL have a **90-day** retention.
- Looker Studio dashboards SHALL NOT query the raw `cloud_armor_lb_logs` table. Analytics reports SHALL use truncated IP data from `frontend_tracking_logs`.

### 2.4 Embedding Cache Privacy (DM-011)

- **No search strings are stored** in the `embedding_cache` collection — only UUID v5 identifiers and embedding vectors.
- UUID v5 document IDs are cryptographic hashes (SHA-1 based) and cannot be reversed to recover the original query text.

## 3. Architectural Absolutes

### 3.1 API Surface (AD-004, BE-API)

- **The API is GET-only with a single POST exception.** `POST` is accepted only on the `/t` endpoint (tracking and error reporting).
- `OPTIONS` is accepted on all endpoints for CORS preflight handling.
- Any other HTTP method on any endpoint SHALL return `405 Method Not Allowed` (SEC-003).
- **THE SYSTEM SHALL NEVER return HTTP `500` to clients.** (Restated for emphasis.)

### 3.2 Fixed Page Size (AD-005, BE-API-002)

- All paginated endpoints SHALL return exactly **10 items per page**. This is fixed and not configurable by the client.

### 3.3 Dual Database Architecture (AD-018)

- **Firestore Enterprise** (MongoDB compatibility mode): Stores all application data. Accessed via the MongoDB Go driver.
- **Firestore Native Mode**: Stores vector embeddings for semantic search only. Accessed via the Firestore Go SDK.
- These are two separate databases in the same GCP project. **Do not mix drivers or purposes.**

### 3.4 Region Constraint

- **All GCP resources** SHALL be deployed in `asia-southeast1`.

### 3.5 Domain Separation (AD-007)

- **Frontend**: `tjmonsi.com` (Firebase Hosting + Firebase Functions).
- **Backend API**: `api.tjmonsi.com` (Cloud Run behind Load Balancer + Cloud Armor).

### 3.6 Content Pipeline Separation (AD-008)

- Content (articles, front page, social links, external references) is managed through a **separate Git repository**. The content pipeline's implementation is out of scope.
- The public-facing API remains **read-only** (`GET`-only, except `POST /t` for tracking).

### 3.7 No Firestore for Tracking/Error Data

- **THE SYSTEM SHALL NOT write tracking or error report data to Firestore.** Structured logging to stdout is the sole persistence mechanism; Cloud Logging routes entries to BigQuery.

## 4. Rate Limiting and Banning Absolutes (SEC-002)

- A single rate limit of **60 requests per minute per IP** is enforced by Cloud Armor at the load balancer level for all clients and all endpoints.
- **THE SYSTEM SHALL NOT apply separate rate limits to bots** — the single 60 req/min limit applies to all clients equally.
- `OPTIONS` preflight requests SHALL NOT be subject to application-level ban checks.

### 4.1 Progressive Banning Tiers

| Condition | Action | Response Code |
|---|---|---|
| Rate limit exceeded (offenses 1–5) | Return rate limit response | `429` |
| 5 offenses within 7 days | Block for 30 days | `403` |
| 2 offenses within 7 days after 30-day ban expires | Block for 90 days | `404` |
| 2 offenses within 7 days after 90-day ban expires | Block indefinitely | `404` |

- 90-day and indefinite blocks return `404` (maximum opacity — attacker does not know they are banned).
- Ban expiry information is included in `403` responses only.

## 5. Authentication Absolutes (SEC-003A, SEC-006)

### 5.1 POST /t Authentication

- `POST /t` requires a valid JWT in the `token` field of the JSON request body.
- JWT algorithm: **HMAC-SHA256**.
- JWT expiry: **5 minutes**.
- Clock skew tolerance: **60 seconds**.
- If the `token` field is missing, empty, or validation fails, return **`403 Forbidden`**.

### 5.2 CORS

- Allowed origin: **`https://tjmonsi.com`** (and development environment equivalents).
- Allowed methods: `GET, POST, OPTIONS`.

## 6. Infrastructure Absolutes

### 6.1 Infrastructure as Code (AD-023, INFRA-016)

- All GCP resources (except manually bootstrapped ones) SHALL be managed via **Terraform**.
- Terraform state is stored remotely in a GCS bucket with versioning and locking.
- The Terraform state bucket SHALL NOT be publicly accessible. Only the Terraform service account SHALL have write access (SEC-012).

### 6.2 Container Security (INFRA-003)

- Docker base images SHALL use **minimal, hardened images** (distroless preferred) (SEC-008).
- Docker images SHALL be scanned for vulnerabilities before deployment.
- The container SHALL run as a **non-root user** (`jack`, UID 10001).

### 6.3 Networking (INFRA-009, AD-011)

- Production uses **Direct VPC Egress** for Cloud Run and Cloud Functions — no Serverless VPC Access Connectors, no NAT router.
- Egress SHALL be restricted to Google Cloud API endpoints only (via Private Google Access).

### 6.4 Workload Identity Federation

- All CI/CD pipelines (Terraform, application deployment, content pipeline) authenticate via **Workload Identity Federation**. No long-lived service account keys in CI/CD.
- Each WIF pool/provider is scoped to a specific repository via attribute conditions.

## 7. Vector Search Absolutes (AD-021, DM-012)

- Embedding model: **Gemini `gemini-embedding-001`** via Vertex AI.
- Embedding dimensions: exactly **2048** (`output_dimensionality=2048`).
- Vectors MUST be **L2-normalized** before storage and comparison.
- Distance measure: **cosine**.
- Cosine distance threshold: **0.35** (results with distance > 0.35 are excluded).
- Document IDs in vector collections MUST match the corresponding `_id` in Firestore Enterprise for 1:1 mapping.
- When `q` parameter is present, the system fetches up to **500** candidate document IDs from Firestore Native via `findNearest(limit: 500)`.

## 8. Security Headers Absolutes (SEC-005)

Security headers SHALL be applied on **both** the frontend and the backend on all responses:

| Header | Value |
|---|---|
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY` |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` |
| `Content-Security-Policy` | Backend: `default-src 'none'`; Frontend: scoped to `'self'` + `api.tjmonsi.com` |

## 9. Dependency and Deployment Security (SEC-008, INFRA-020)

- Go dependencies SHALL be audited with **`govulncheck`** in CI/CD.
- Frontend dependencies SHALL be audited with **`npm audit`** in CI/CD.
- Docker images SHALL be scanned with vulnerability scanners (e.g., **Trivy**) before deployment.

## 10. Input Validation Absolutes (SEC-001)

- Search queries exceeding **300 characters**: reject with `400`.
- Tags per request: maximum **10**; reject with `400` if exceeded.
- `POST /t` payloads exceeding **100 KB**: reject with `413 Payload Too Large`.
- Slug pattern: `^[a-z0-9]+(?:-[a-z0-9]+)*-\d{4}-\d{2}-\d{2}-\d{4}\.md$`; non-matching slugs return `404`.
- All tag values MUST be **lowercase**.
- `date_from` must be on or before `date_to`; violation returns `400`.
